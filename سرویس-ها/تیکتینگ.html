سند چشم انداز و محدوده پروژه

1. معرفی

هدف از این سند ارائه چشم انداز و محدوده روشن برای توسعه میکروسرویس تیکتینگ است. اهداف، مقاصد و محدودیت‌های پروژه را مشخص می‌کند و اطمینان می‌دهد که همه ذینفعان درک مشترکی از هدف و قابل تحویل آن دارند.

2. چشم انداز

چشم انداز میکروسرویس پشتیبانی تیکتینگ ارائه یک سیستم متمرکز برای مدیریت تینگ های پشتیبانی مشتری است. این میکروسرویس به کاربران امکان می‌دهد تیکت بسازند، برای مدیران پیام ارسال کنند و وضعیت تیکت‌های خود را پیگیری کنند. مدیران این امکان را خواهند داشت که به تیکت ها پاسخ دهند، وضعیت تیکت را تغییر دهند و پیوست هایی مانند متن، تصاویر و صدا را مدیریت کنند.

2. اهداف پروژه

اهداف اصلی میکروسرویس پشتیبانی تیکتینگ عبارتند از:
  • کاربران را قادر می سازد تا بلیط های پشتیبانی را با جزئیات مرتبط ایجاد کنند.
  • تسهیل ارتباط بین کاربران و مدیران از طریق پیام.
  • به مدیران امکان مشاهده و پاسخگویی به تیکت ها را بدهید.
  • به مدیران اجازه دهید وضعیت‌های تیکت را مدیریت کنند (به عنوان مثال، در انتظار، در حال انجام، به اتمام رسیده).
  • پشتیبانی از ضمیمه انواع مختلف فایل از جمله متن، تصویر و صدا.
  • از امنیت و محرمانه بودن اطلاعات کاربر اطمینان حاصل کنید.
  • ارائه یک معماری میکروسرویس قابل اعتماد و مقیاس پذیر.

3. Scope of the Project

The scope of the Support Ticketing microservice includes the following features and functionalities:
  • Ticket Creation:
    • Users can create support tickets with relevant information, including a title, description, and any necessary attachments.
    • Each ticket will be assigned a unique identifier for tracking purposes.
  • Ticket Messaging:
    • Users and administrators can exchange messages within the context of a ticket.
    • Messages can include text, images, and voice attachments.
    • Notifications will be sent to users and administrators when new messages are received.
  • Administrator Actions:
    • Administrators have the ability to view and respond to tickets.
    • They can change ticket statuses (e.g., pending, in progress, resolved) based on the ticket's progress.
    • Administrators can escalate tickets to higher levels of support if necessary.
  • Attachment Handling:
    • The microservice should support the upload, storage, and retrieval of text, image, and voice attachments.
    • Attachments should be accessible within the context of their associated tickets.
  • Security and Data Privacy:
    • User data and attachments should be securely stored and transmitted.
    • Authentication and authorization mechanisms should be implemented to control access to the microservice's functionalities.
    • User privacy and confidentiality should be maintained throughout the system.

4. ذینفعان

  • کاربران: افرادی که نیاز به بالا بردن بلیط های پشتیبانی و ارتباط با مدیران دارند.
  • مدیران: افرادی که مسئول رسیدگی و پاسخگویی به بلیط های پشتیبانی هستند.
  • تیم توسعه: تیمی که مسئول طراحی، توسعه و استقرار میکروسرویس پشتیبانی بلیط است.
  • مدیریت: ذینفعان پروژه و تصمیم گیرندگانی که بر اجرای میکروسرویس نظارت دارند.

5. محدودیت ها

پشته فناوری: میکروسرویس باید با استفاده از فناوری ها و چارچوب های مشخص شده ساخته شود.
یکپارچه سازی: میکروسرویس باید طوری طراحی شود که در صورت لزوم با سیستم ها و API های موجود یکپارچه شود.

6. مفروضات

کاربران به دستگاه های لازم و اتصال اینترنتی برای دسترسی به میکروسرویس دسترسی خواهند داشت.
زیرساخت ها و منابع کافی برای میزبانی و استقرار میکروسرویس در دسترس خواهد بود.
هر گونه سرویس خارجی یا API ضروری در دسترس و قابل دسترسی خواهد بود.
میکروسرویس با استفاده از یک زبان برنامه نویسی یا فریم ورک خاص توسعه خواهد یافت.

7. خطرات

عدم پذیرش یا پذیرش ناکافی سیستم تیکتینگ پشتیبانی توسط کاربر.
آسیب‌پذیری‌های امنیتی که منجر به دسترسی غیرمجاز یا نقض داده‌ها می‌شود.
مقیاس پذیری ناکافی برای مدیریت افزایش حجم بلیط و بارهای کاربر.
چالش های یکپارچه سازی با سیستم های موجود یا خدمات شخص ثالث.
تأخیرهای بالقوه یا محدودیت های منابع که بر جدول زمانی پروژه تأثیر می گذارد.

2. System Requirements Specification

2.1 User Management

  • FR 2.1.2: Users should be able to authenticate themselves using their credentials to access the microservice. (Using provided JWT Token with Identity microservice)
  • FR 2.1.3: The microservice should support different user roles, including regular users and administrators.

2.2 Ticket Management

  • FR 2.2.1: Users should be able to create support tickets by providing a title, description, and any relevant attachments.
  • FR 2.2.2: Each ticket should have a unique identifier for tracking and reference purposes.
  • FR 2.2.3: Users should be able to view and edit their own tickets.
  • FR 2.2.4: Administrators should have the ability to view all tickets and filter them based on different criteria (e.g., status, priority).

2.3 Messaging

  • FR 2.3.1: Users and administrators should be able to exchange messages within the context of a ticket.
  • FR 2.3.2: Messages should support various content types, including plain text, images, and voice recordings.
  • FR 2.3.3: Notifications should be sent to users and administrators when new messages are received.

2.4 Ticket Status Management

  • FR 2.4.1: Administrators should be able to change the status of a ticket (e.g., pending, in progress, resolved).
  • FR 2.4.2: Users should be able to track the status of their tickets and view any updates made by administrators.

2.5 Attachment Handling

  • FR 2.5.1: The microservice should support the upload, storage, and retrieval of text, image, and voice attachments.
  • FR 2.5.2: Attachments should be associated with their respective tickets and accessible within the ticket view.

3. Non-functional Requirements

3.1 Performance

  • NFR 3.1.1: The microservice should be able to handle a large number of concurrent users without significant performance degradation.
  • NFR 3.1.2: The response time for essential operations, such as ticket creation and message retrieval, should be within acceptable limits.
  • NFR 3.1.3: The system should be scalable to accommodate increasing user loads and ticket volumes.

3.2 Security

  • NFR 3.2.1: User data and attachments should be securely stored and transmitted using appropriate encryption mechanisms.
  • NFR 3.2.2: The microservice should implement authentication and authorization mechanisms to control access to its functionalities.
  • NFR 3.2.3: User privacy and confidentiality should be maintained throughout the system.

3.3 Reliability

  • NFR 3.3.1: The microservice should have a high level of availability, with minimal downtime for maintenance and updates.
  • NFR 3.3.2: Data integrity should be ensured through appropriate backup and recovery mechanisms.
  • NFR 3.3.3: The system should be resilient to failures and errors, with proper error handling and graceful degradation.

3.4 Usability

  • NFR 3.4.1: The user interface should be intuitive, user-friendly, and accessible.
  • NFR 3.4.2: The microservice should provide clear and informative error messages to assist users in resolving issues.
  • NFR 3.4.3: The system should support multiple languages to cater to a diverse user base.

  • The project must be completed within the agreed-upon timeframe and budget.

Architecture and Design Document

1. Introduction

The Architecture and Design Document provides an overview of the architecture, components, and design considerations for the Support Ticketing microservice. It outlines the system's high-level structure, describes the key components, and discusses the design decisions made to achieve the desired functionality and requirements.

2. Architecture Overview

The Support Ticketing microservice will follow a microservices architecture pattern. It will consist of the following components:

2.1 User Management Service

  • Manages user roles and permissions.

2.2 Ticket Service

  • Handles ticket creation, retrieval, and management.
  • Stores ticket details, including status and associated user information.

2.3 Messaging Service

  • Enables users and administrators to exchange messages within the context of a ticket.
  • Stores and retrieves messages, including attachments.

2.4 Admin Service

  • Provides functionality for administrators to view and respond to tickets.
  • Manages ticket status changes.

2.5 Attachment Service

  • Handles the upload, storage, and retrieval of text, image, and voice attachments.
  • Associates attachments with their respective tickets.

2.6 Notification Service

  • Sends notifications to users and administrators for new messages and ticket updates.

3.2 API Gateway

  • Acts as a single entry point for incoming requests to the microservice.
  • Routes requests to the appropriate internal services.

3.3 Service Orchestration

  • Coordinates the interaction between different microservices and manages the flow of requests and responses.
  • Ensures data consistency and integrity across services.

3.4 Data Storage

  • Utilizes a database system to store and retrieve persistent data.
  • Stores ticket information, user details, messages, and attachments.

3.5 External Integrations

  • Interfaces with external services or APIs, if necessary, for authentication, messaging, and attachment handling.

4. Design Considerations

4.1 Scalability

  • The microservice architecture should be designed to scale horizontally to handle increased user loads and ticket volumes.
  • Services should be deployed in a distributed manner, allowing for load balancing and auto-scaling as needed.

4.2 Security

  • User authentication and authorization should be implemented to ensure proper access control.
  • Sensitive data, including user information and attachments, should be stored securely and encrypted during transmission.

4.3 Performance

  • Caching mechanisms should be employed where appropriate to improve response times and reduce database load.
  • Asynchronous messaging and event-driven architecture can be utilized for real-time updates and notifications.

4.4 Resilience and Fault Tolerance

  • Services should be designed to handle failures gracefully and recover from errors.
  • Proper error handling and logging should be implemented to aid in debugging and issue resolution.

4.5 Deployment and Infrastructure

  • The microservice should be containerized using technologies like Docker for easy deployment and scalability.
  • Infrastructure should be provisioned to support the microservice, including load balancers, auto-scaling groups, and database clusters.

API Documentation

Base URL

https://api.example.com/ticketing/v[version]

Authentication

The Support Ticketing microservice uses JSON Web Tokens (JWT) for authentication. To authenticate a request, include the JWT token in the Authorization header using the Bearer scheme.
Authorization: Bearer <JWT_TOKEN>

Endpoints

1. Create a Ticket

Endpoint: /tickets
Method: POST
Request:
{
  "title": "Example Ticket",
  "description": "This is an example ticket",
  "priority": 1
}
Response:
{
  "id": "123456",
  "title": "Example Ticket",
  "description": "This is an example ticket",
  "status": "pending",
  "createdAt": "2023-07-17T10:00:00Z",
  "updatedAt": "2023-07-17T10:00:00Z"
}

2. Get a ticket

Endpoint: /tickets/{ticketId}
Method: GET
Response: 
{
  "id": "123456",
  "title": "Example Ticket",
  "description": "This is an example ticket",
  "status": "pending",
  "createdAt": "2023-07-17T10:00:00Z",
  "updatedAt": "2023-07-17T10:00:00Z"
}

3. Get all user tickets

Endpoint: /tickets/
Filterings: Status/Title/Date
Method: GET
Response:

		  "title": "Example Ticket",
		  "description": "This is an example ticket",
		  "status": "pending",
		  "createdAt": "2023-07-17T10:00:00Z",
		  "updatedAt": "2023-07-17T10:00:00Z",
		}
	]
}

4. Add a Message to a Ticket

Endpoint: /tickets/{ticketId}/messages

Database Schema

Tickets Table:
Column Type Description
id INTEGER Ticket ID
title VARCHAR(100) Ticket title
description TEXT Ticket description
status VARCHAR(20) Ticket status
created_at TIMESTAMP Date and time of ticket creation
updated_at TIMESTAMP Date and time of ticket update
Messages Table:
Column Type Description
id INTEGER Message ID
ticket_id INTEGER ID of the associated ticket
content TEXT Message content
created_at TIMESTAMP Date and time of message creation
author_id INTEGER ID of the message author
author_username VARCHAR(20) Username of author
author_ip VARCHAR(15) Ip of author, null if admin
Attachments Table:
Column Type Description
id INTEGER Attachment ID
ticket_id INTEGER ID of the associated ticket
type VARCHAR(20) Attachment type (e.g., image, voice)
url VARCHAR(255) URL of the attachment
message_id INTEGER ID of the associated message
