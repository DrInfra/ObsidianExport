سند چشم انداز و محدوده پروژه

1. معرفی

هدف از این سند ارائه چشم انداز و محدوده روشن برای توسعه میکروسرویس هویت است. اهداف، مقاصد و محدودیت‌های پروژه را مشخص می‌کند و اطمینان می‌دهد که همه ذینفعان درک مشترکی از هدف و قابل تحویل آن دارند.

2. بررسی اجمالی پروژه

2.1 شرح پروژه

هدف میکروسرویس هویت ارائه یک راه حل قوی و مقیاس پذیر برای احراز هویت کاربر، ثبت نام، مدیریت مجوز و مدیریت نقش است. این به عنوان یک مؤلفه مرکزی در سیستم ما عمل می کند و کنترل دسترسی ایمن و مدیریت کاربر را در همه برنامه ها و خدمات ممکن می کند.

2.2 اهداف پروژه

اهداف اولیه میکروسرویس هویت به شرح زیر است:
ثبت نام و احراز هویت کاربر را با مکانیسم های ذخیره و بازیابی رمز عبور ایمن.
ارائه یک سیستم مدیریت مجوز منعطف و دقیق، به مدیران اجازه می دهد تا مجوزها را به کاربران و نقش ها تعریف و اختصاص دهند.
از کنترل دسترسی مبتنی بر نقش (RBAC) پشتیبانی می کند که مجوز کارآمد و ثابت را در بین برنامه ها و خدمات تسهیل می کند.
یک رابط کاربر پسند و API برای ادغام یکپارچه با سایر سیستم ها و خدمات ارائه دهید.
از در دسترس بودن، مقیاس پذیری و تحمل خطا بالا برای مدیریت پایگاه کاربر رو به رشد و افزایش تقاضا اطمینان حاصل کنید.

2.3 ذینفعان

ذینفعان کلیدی پروژه میکروسرویس هویت عبارتند از:
تیم توسعه: مسئول طراحی، پیاده سازی و نگهداری میکروسرویس است.
مالک محصول: اطلاعات لازم را در مورد نیازهای کسب و کار ارائه می دهد و ویژگی ها و پیشرفت ها را اولویت بندی می کند.
مدیران سیستم: از میکروسرویس برای مدیریت کاربران، مجوزها و نقش ها استفاده کنید.
توسعه دهندگان برنامه: برای احراز هویت و مجوز دادن به کاربران در برنامه هایشان، با میکروسرویس یکپارچه شوید.
کاربران نهایی: با میکروسرویس از طریق رابط کاربری یا API برای ثبت نام، ورود به سیستم و مدیریت مجوزهای آنها تعامل داشته باشید.

3. محدوده پروژه

3.1 ویژگی های In-Scope

ثبت نام کاربر: به کاربران اجازه می دهد تا با ارائه اطلاعات مورد نیاز و اعتبارسنجی آن، حساب های جدید ایجاد کنند.
ورود کاربر: کاربران را قادر می سازد تا احراز هویت و دسترسی ایمن به سیستم را داشته باشند.
مدیریت رمز عبور: امکان بازنشانی یا تغییر رمز عبور را برای کاربران فراهم می کند.
مدیریت مجوز: به مدیران اجازه می دهد تا مجوزها را برای اقدامات و منابع مختلف تعریف و مدیریت کنند.
مدیریت نقش: از ایجاد، واگذاری و حذف نقش‌ها به کاربران پشتیبانی می‌کند و مجوز کارآمد را بر اساس نقش‌ها ممکن می‌سازد.
مدیریت پروفایل کاربر: به کاربران اجازه می دهد اطلاعات شخصی خود را به روز کنند و تنظیمات برگزیده خود را مدیریت کنند.

3.2 ویژگی های Out-Scope

قابلیت‌های خاص کاربر مربوط به برنامه‌ها یا سرویس‌های فردی (به عنوان مثال، سفارشی‌سازی نمایه در یک برنامه خاص).
ادغام ورود به سیستم اجتماعی (به عنوان مثال، با فیس بوک یا گوگل وارد شوید).
اجرای احراز هویت چند عاملی (MFA).

3.3 محدودیت ها و مفروضات

این میکروسرویس با استفاده از یک استک فناوری مدرن و مقیاس‌پذیر که با زیرساخت‌های موجود و بهترین شیوه‌های سازمان هماهنگ است، توسعه می‌یابد. (بخدا مدیونید فکر کنید این تیکه رو از چت جی پی تی کپی کردم!)
ادغام با سیستم‌ها و سرویس‌های موجود از طریق APIها و پروتکل‌های کاملاً تعریف‌شده به دست خواهد آمد.
میکروسرویس در زیرساخت ابری ترجیحی سازمان یا سرورهای داخلی میزبانی خواهد شد.
میکروسرویس باید با مقررات مربوط به امنیت و حریم خصوصی (مانند GDPR، HIPAA) مطابقت داشته باشد.

5. ریسک های پروژه

خطرات شناسایی شده مرتبط با پروژه باید مستند شده و در یک برنامه مدیریت ریسک جداگانه مورد رسیدگی قرار گیرد. خطرات ممکن است شامل موارد زیر باشد:
چالش های یکپارچه سازی با سیستم ها و خدمات موجود
مسائل مقیاس پذیری و عملکرد
آسیب پذیری های امنیتی یا نقض داده ها.
تغییرات در الزامات مقرراتی یا انطباق.

مشخصات سیستم مورد نیاز

1. معرفی

سند مشخصات نیازمندیهای سیستم، الزامات عملکردی و غیرعملکردی میکروسرویس هویت را تشریح می کند. ویژگی‌ها، رابط‌ها، معیارهای عملکرد و محدودیت‌هایی که میکروسرویس باید به آن‌ها پایبند باشد را تعریف می‌کند.

2. الزامات عملکردی

2.1 ثبت نام کاربر

میکروسرویس باید به کاربران اجازه دهد تا با ارائه اطلاعات زیر حساب‌های جدیدی ایجاد کنند:
نام کاربری (شناسه منحصر به فرد)
آدرس ایمیل (محصول)
رمز عبور (ذخیره شده و رمزگذاری شده ایمن)
فرآیند ثبت نام باید اطلاعات ارائه شده را تایید کند و از منحصر به فرد بودن آن اطمینان حاصل کند.
پس از ثبت نام موفقیت آمیز، میکروسرویس باید یک ایمیل تأیید برای تأیید آدرس ایمیل کاربر ایجاد کند.

2.2 ورود کاربر

میکروسرویس باید یک مکانیسم ورود امن برای کاربران ثبت نام شده فراهم کند.
کاربران باید بتوانند با استفاده از نام کاربری/ایمیل و رمز عبور خود احراز هویت کنند.
ورود موفقیت آمیز باید یک نشانه احراز هویت (به عنوان مثال، JWT) ایجاد کند تا برای درخواست های API بعدی استفاده شود.

2.3 مدیریت رمز عبور

کاربران باید بتوانند پسوردهای خود را در صورت فراموشی بازنشانی کنند.
میکروسرویس باید یک قابلیت بازنشانی رمز عبور ارائه دهد که یک ایمیل بازنشانی رمز عبور را به آدرس ایمیل ثبت شده کاربر ارسال می کند.
ایمیل بازنشانی رمز عبور باید حاوی یک پیوند امن باشد که به کاربر اجازه می دهد رمز عبور جدیدی تنظیم کند.

2.4 مدیریت مجوز

مدیران باید بتوانند مجوزهایی را برای اقدامات و منابع مختلف در سیستم تعریف کنند.
میکروسرویس باید عملیات CRUD (ایجاد، خواندن، به‌روزرسانی، حذف) را برای مدیریت مجوزها ارائه دهد.
مجوزها باید به تک تک کاربران یا نقش ها اختصاص داده شوند.

2.5 مدیریت نقش

مدیران باید بتوانند نقش ها را ایجاد کرده و مجوزها را به آنها اختصاص دهند.
میکروسرویس باید از عملیات CRUD برای مدیریت نقش ها پشتیبانی کند.
نقش ها باید به تک تک کاربران قابل واگذاری باشد.

2.6 مدیریت نمایه کاربر

کاربران باید بتوانند اطلاعات پروفایل خود را مشاهده و به روز کنند.
میکروسرویس باید APIهایی را برای کاربران فراهم کند تا اطلاعات شخصی خود مانند نام، ایمیل و عکس نمایه را به روز کنند.

3. الزامات غیر عملکردی

3.1 عملکرد

میکروسرویس باید حجم بالایی از درخواست‌های همزمان کاربر را به نحو احسن انجام دهد.
زمان پاسخ برای احراز هویت کاربر، ثبت نام و عملیات مدیریت مجوز/نقش باید در محدوده قابل قبول باشد.

3.2 امنیت

گذرواژه‌های کاربر باید با استفاده از الگوریتم‌های رمزگذاری استاندارد صنعتی ذخیره و رمزگذاری شوند.
تمام ورودی های کاربر باید برای جلوگیری از آسیب پذیری های امنیتی رایج (به عنوان مثال، تزریق SQL، اسکریپت بین سایتی) اعتبارسنجی و پاکسازی شود.
مکانیسم‌های احراز هویت و مجوز باید برای اطمینان از کنترل دسترسی ایمن اجرا شوند.
دسترسی به داده های حساس و عملیات ممتاز باید فقط به پرسنل مجاز محدود شود.

3.3 مقیاس پذیری

میکروسرویس باید به گونه ای طراحی شود که به صورت افقی برای مدیریت بارهای کاربر افزایش می یابد. (مقیاس پذیری افقی)
باید از استقرار توزیع شده و متعادل سازی بار برای اطمینان از عملکرد بهینه پشتیبانی کند.

3.4 قابلیت اطمینان

میکروسرویس باید بسیار در دسترس و در برابر خرابی ها مقاوم باشد.
این باید شامل مکانیسم های مدیریت خطا و بازیابی مناسب برای به حداقل رساندن زمان خرابی باشد.

3.5 یکپارچه سازی

میکروسرویس باید Rest/Restful API ها و پروتکل های کاملاً تعریف شده ای را ارائه دهد تا امکان ادغام یکپارچه با سایر سیستم ها و خدمات فراهم شود.
ادغام با دایرکتوری های کاربر موجود یا ارائه دهندگان هویت، در صورت وجود، باید پشتیبانی شود.

3.6 انطباق

میکروسرویس باید با مقررات مربوط به امنیت، حریم خصوصی و حفاظت از داده ها مانند GDPR یا HIPAA مطابقت داشته باشد.

4. محدودیت ها و مفروضات

میکروسرویس از یک پایگاه داده (رابطه ای یا NoSQL) برای ذخیره داده ها، مجوزها و نقش های کاربر استفاده می کند.
میکروسرویس بر روی (زیرساخت های ابری یا سرورهای داخلی On-Promise) میزبانی خواهد شد.
ادغام با سایر سیستم ها یا خدمات از طریق (پروتکل ها یا API ها) حاصل خواهد شد.

5. وابستگی ها

میکروسرویس ممکن است به اجزای زیرساخت موجود، مانند پایگاه‌های داده، صف‌های پیام یا سرویس‌های احراز هویت بستگی داشته باشد.

6. معیارهای پذیرش

میکروسرویس باید بر اساس معیارهای پذیرش از پیش تعریف شده آزمایش شود تا از انطباق با الزامات مشخص شده اطمینان حاصل شود.
موارد تست باید تمام الزامات کاربردی و غیرعملکردی ذکر شده در این سند را پوشش دهند.
این سند مشخصات نیازمندیهای سیستم یک نمای کلی از الزامات عملکردی و غیرعملکردی برای میکروسرویس هویت ارائه می دهد. این به عنوان یک مرجع برای تیم توسعه عمل می کند تا اطمینان حاصل شود که همه ویژگی ها و محدودیت ها به درستی اجرا می شوند.

سند معماری و طراحی

1. معرفی

سند معماری و طراحی یک نمای کلی از معماری و انتخاب های طراحی برای میکروسرویس هویت ارائه می دهد. اجزای سیستم، تعاملات آنها و ساختار کلی میکروسرویس را تشریح می کند.

2. معماری سیستم

2.1 سبک معماری

میکروسرویس هویت از معماری میکروسرویس پیروی می کند که مدولار بودن، مقیاس پذیری و استقرار مستقل خدمات را ارتقا می دهد. از طریق APIهای تعریف شده با سایر ریزسرویس ها و سیستم های خارجی تعامل خواهد داشت.

2.2 اجزای سطح بالا

میکروسرویس شامل اجزای سطح بالا زیر است:
ز API Gateway: به عنوان نقطه ورود برای درخواست های خارجی عمل می کند و احراز هویت و مسیریابی به میکروسرویس های مناسب را مدیریت می کند.
سرویس احراز هویت: مسئول احراز هویت کاربر، از جمله عملکردهای مدیریت ثبت نام، ورود به سیستم و رمز عبور است.
سرویس مجوز و نقش: ایجاد، واگذاری و مدیریت مجوزها و نقش ها را مدیریت می کند.
سرویس پروفایل کاربر: اطلاعات نمایه کاربر، از جمله به روز رسانی اطلاعات شخصی و تصاویر نمایه را کنترل می کند.
ذخیره سازی داده ها: داده های کاربر، مجوزها، نقش ها و اطلاعات مرتبط را ذخیره می کند. این ممکن است از یک پایگاه داده رابطه ای یا NoSQL بسته به نیازهای خاص استفاده کند.

2.3 ملاحظات امنیتی

احراز هویت کاربر با استفاده از پروتکل‌های استاندارد صنعتی، مانند OAuth یا JWT، برای اطمینان از کنترل دسترسی ایمن اجرا می‌شود.
گذرواژه ها با استفاده از یک الگوریتم هش قوی به صورت ایمن ذخیره و رمزگذاری می شوند.
دسترسی به API ها و عملیات حساس از طریق مکانیسم های مجوز مناسب محافظت می شود.
اعتبار سنجی ورودی و کدگذاری خروجی برای جلوگیری از آسیب پذیری های امنیتی رایج پیاده سازی خواهد شد.

3. طراحی پایگاه داده

میکروسرویس از یک پایگاه داده برای ذخیره اطلاعات کاربر، مجوزها، نقش ها و داده های مرتبط استفاده می کند. فناوری پایگاه داده خاص (رابطه ای یا NoSQL) بر اساس الزامات پروژه و ملاحظات مقیاس پذیری انتخاب خواهد شد.
طرح پایگاه داده شامل جداول/موارد زیر خواهد بود:
کاربران: اطلاعات کاربر از جمله نام کاربری، ایمیل، رمز عبور و جزئیات نمایه را ذخیره می کند.
مجوزها: لیست مجوزهای موجود را در سیستم ذخیره می کند.
زRoles: نقش های تعریف شده در سیستم، از جمله مجوزهای مربوط به آنها را ذخیره می کند.
زUserRoles: نشان دهنده رابطه چند به چند بین کاربران و نقش ها است.
زUserPermissions: نشان دهنده رابطه چند به چند بین کاربران و مجوزها است.

4. معماری استقرار

4.1 محیط میزبانی

میکروسرویس بر روی (زیرساخت‌های ابری یا سرورهای داخلی) با پیروی از Best practices برای مقیاس‌پذیری، در دسترس بودن و عملکرد مستقر می‌شود. محیط میزبانی خاص بر اساس ترجیحات و الزامات سازمانی تعیین خواهد شد.

4.2 استراتژی استقرار

این میکروسرویس با استفاده از (فناوری کانتینری‌سازی)، مانند داکر، کانتینری‌سازی می‌شود تا از سازگاری و قابلیت حمل در محیط‌های مختلف اطمینان حاصل شود. این ابزار با استفاده از(ابزار ارکستریشن) که امکان مقیاس‌بندی آسان، متعادل‌سازی بار و تحمل خطا را فراهم می‌کند، مستقر می‌شود.

4.3 الزامات زیرساخت

زیرساخت برای استقرار میکروسرویس شامل موارد زیر خواهد بود:
منابع محاسباتی (ماشین های مجازی یا کانتینرها) برای میزبانی اجزای میکروسرویس.
پیکربندی شبکه برای ارتباط داخلی بین میکروسرویس ها و دسترسی خارجی به API ها.
راه حل های ذخیره سازی برای پایداری پایگاه داده و ذخیره سازی فایل (در صورت نیاز).
مکانیسم های نظارت و ثبت برای ردیابی عملکرد سیستم و عیب یابی مشکلات.

5. نقاط ادغام

میکروسرویس هویت ممکن است با سیستم ها یا خدمات خارجی زیر یکپارچه شود:
زAPI Gateway: برای مسیریابی درخواست، تعادل بار و امنیت با یک دروازه API یا پروکسی معکوس ادغام می شود.
ارائه دهندگان احراز هویت: در صورت نیاز، با ارائه دهندگان احراز هویت خارجی (به عنوان مثال، ارائه دهندگان OAuth) برای ورود به سیستم اجتماعی یا احراز هویت فدرال یکپارچه شوید.
سیستم‌های ثبت و نظارت: با سیستم‌های ثبت و نظارت ادغام می‌شوند تا گزارش‌ها، معیارها و داده‌های عملکرد سیستم را ضبط کنند.

6. استراتژی تست

یک استراتژی آزمایش جامع برای اطمینان از کیفیت و قابلیت اطمینان میکروسرویس به کار گرفته خواهد شد. باید شامل انواع تست زیر باشد:
تست واحد: تک تک اجزا و عملکردها را به صورت مجزا آزمایش کنید.
تست یکپارچه سازی: ادغام بین اجزای میکروسرویس و وابستگی های خارجی را آزمایش کنید.
تست سرتاسری: کل عملکرد سیستم را از طریق موارد استفاده نماینده آزمایش کنید.
تست عملکرد: ارزیابی عملکرد میکروسرویس تحت شرایط بار مختلف.
تست امنیتی: آسیب پذیری های امنیتی بالقوه را شناسایی و برطرف کنید.

7. خطرات و استراتژی های کاهش

هر گونه خطر یا چالش مرتبط با معماری و طراحی میکروسرویس را شناسایی و مستند کنید. برای هر ریسک، یک استراتژی کاهش برای رسیدگی یا به حداقل رساندن تأثیر آن ارائه کنید.

اسناد API

معرفی

اسناد API اطلاعات دقیقی در مورد نقاط پایانی، فرمت‌های درخواست/پاسخ، و مکانیسم‌های احراز هویت برای میکروسرویس هویت ارائه می‌کند. این سند به عنوان راهنمای توسعه دهندگان و ادغام کنندگانی است که مایل به تعامل با API میکروسرویس هستند.

پایه URL

پایه URL برای دسترسی به API microservice هویت این است:
https://api.example.com/Identity/v[version]

احراز هویت

میکروسرویس هویت از [مکانیسم احراز هویت] برای کنترل دسترسی ایمن استفاده می کند. برای تأیید اعتبار درخواست‌ها، هدر زیر را در هر تماس API وارد کنید:
Authorization: Bearer [access_token]
پارامتر «[access_token]» را با رمز احراز هویت که از طریق فرآیند ورود به سیستم به دست آمده است، جایگزین کنید.

Endpoints

1. User Registration

Endpoint: /register
Method: POST
Description: Create a new user account.
{
  "username": "john_doe",
  "email": "johndoe@example.com",
  "password": "secretpassword"
}
Response:
{
  "message": "User registration successful.",
  "user_id": "123456789"
}

2. User Login

Endpoint: /login
Method: POST
Description: Authenticate a user and obtain an access token.
Request:
{
  "username": "john_doe",
  "password": "secretpassword"
}
Response:
{
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...", // in JWT
  "refresh_token": "eyWQEnnRqweqsad..."
  "expires_in": 3600, // Seconds
  "refresh_token_expired": false // Always false for login - no need to check
}

3. Refresh token:

Endpoint: /login
Method: POST
Description: Authenticate a user and obtain an access token.
Request:
{
  "refresh_token": "eyWQEnnRqweqsad...",
}
Response:
{
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...", // in JWT
  "refresh_token": "eyWQEnnRqweqsad..."
  "expires_in": 3600, // Seconds
  "refresh_token_expired": false // Can be true, so refresh token can't be used for next time
}

4. Password Reset

Endpoint: /reset-password
Method: POST
Description: Send a password reset email to the user's registered email address.
Request:
{
  "email": "johndoe@example.com"
}
Response:
{
  "message": "Password reset email sent."
}

Database Schema

Table: Users

Column Type Description
user_id INT Unique identifier for the user.
username VARCHAR User's username.
email VARCHAR User's email address.
password VARCHAR Encrypted password for user login.
created_at TIMESTAMP Date and time of user registration.
updated_at TIMESTAMP Date and time of last user profile update.

Table: Permissions

Column Type Description
permission_id INT Unique identifier for the permission.
name VARCHAR Name of the permission.
description VARCHAR Description of the permission.

Table: Roles

Column Type Description
role_id INT Unique identifier for the role.
name VARCHAR Name of the role.
description VARCHAR Description of the role.

Table: UserRoles

Column Type Description
user_id INT Foreign key referencing Users table.
role_id INT Foreign key referencing Roles table.

Table: RolePermissions

Column Type Description
permission_id INT Foreign key referencing Permissions table.
role_id INT Foreign key referencing Roles table.
  • Users: Stores user information, including user ID, username, email, password, and timestamps for registration and profile updates.
  • Permissions: Stores the list of available permissions in the system, with a unique permission ID, name, and description.
  • Roles: Stores the roles defined within the system, with a unique role ID, name, and description.
  • UserRoles: Represents the many-to-many relationship between users and roles. It maps users to the roles they have been assigned.
  • UserPermissions: Represents the many-to-many relationship between users and permissions. It maps users to the permissions they have been granted.
